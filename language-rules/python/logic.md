# Python 代码逻辑规范

## 核心原则

生成和修改 Python 代码时，应采用尽可能简单、易于扩展、易于修改的代码范式。

代码逻辑的优先级：

1. 正确性
2. 可读性
3. 可测试性
4. 可扩展性
5. 性能

只有在性能已经被证明是瓶颈时，才为了性能牺牲可读性。

## 先理解再修改

修改既有项目时，应先理解现有结构：

- 入口文件在哪里。
- 数据从哪里进入，在哪里被校验，在哪里被持久化。
- 业务逻辑放在哪一层。
- 测试如何覆盖当前行为。
- 项目已经使用了哪些框架、工具和命名习惯。

不要在没有理解现有模式的情况下引入新的架构、目录结构或工具链。

## 简单优先

优先选择直接、明确、容易调试的实现。

推荐：

- 普通函数优先于复杂类层级。
- 明确分支优先于隐晦映射。
- 标准库优先于新增第三方依赖。
- 小型数据类或类型模型优先于任意嵌套字典。
- 局部清晰重复优先于过早抽象。

不推荐：

- 为少量代码引入复杂设计模式。
- 为未来不确定需求建立庞大抽象。
- 使用元编程、动态 monkey patch 或隐式注册机制解决普通问题。
- 为了减少几行代码牺牲可读性。

## 单一职责

函数、类和模块都应有清晰职责。

一个函数不应同时完成：

- 读取输入
- 解析数据
- 校验数据
- 执行业务逻辑
- 写入数据库
- 发送网络请求
- 格式化输出

当函数开始混合多个职责时，应拆分为更小的函数，并让每个函数的输入输出清楚可见。

示例：

```python
def create_user(request: CreateUserRequest) -> User:
    normalized_email = normalize_email(request.email)
    ensure_email_is_available(normalized_email)
    user = User(email=normalized_email, name=request.name)
    return user_repository.save(user)
```

这类代码比把所有逻辑写在一个大函数中更容易测试和修改。

## 显式边界

系统边界应明确，包括：

- 外部请求输入
- 配置和环境变量
- 文件系统读写
- 数据库访问
- 网络请求
- 第三方 API
- 用户输入

边界处应进行转换和校验。进入核心业务逻辑后，数据应尽量是可信、结构化、类型明确的。

推荐做法：

- 在入口处解析原始输入。
- 在入口处处理缺失值、非法值和格式错误。
- 在入口处把字典转换为模型或数据类。
- 在核心逻辑中使用明确类型，而不是反复检查原始输入。

## 控制流

控制流应直接、线性、容易跟踪。

- 使用早返回处理无效输入和特殊情况。
- 避免深层嵌套。
- 把复杂条件拆成有名字的布尔变量或函数。
- 对互斥状态使用清晰分支。
- 对复杂流程使用小函数分段表达意图。

示例：

```python
def can_view_project(user: User, project: Project) -> bool:
    if user.is_admin:
        return True

    if project.is_public:
        return True

    return project.has_member(user.id)
```

比深层嵌套更易读，也更容易扩展。

## 数据建模

优先让数据结构表达业务含义。

适合使用 `dict` 的情况：

- 临时局部映射。
- 与 JSON 或外部 API 直接交互的边界层。
- 键集合简单且作用域很小。

不适合长期使用裸 `dict` 的情况：

- 跨多个函数传递。
- 键名依赖隐式约定。
- 值类型复杂或可选字段很多。
- 代表核心业务实体。

这类数据应使用：

- `dataclasses.dataclass`
- `typing.TypedDict`
- `pydantic.BaseModel`
- 项目已有的 ORM model 或 schema

## 依赖管理

新增依赖前应先判断：

- 标准库是否已经能解决问题。
- 项目中是否已有同类依赖。
- 新依赖是否会显著增加安装、构建或运行复杂度。
- 新依赖是否处于维护状态。
- 依赖是否只用于很小的一段逻辑。

如果只是少量简单逻辑，不应为了省几行代码引入新依赖。

新增依赖必须使用 `uv add` 或 `uv add --dev`，并提交对应的依赖文件变化。

## 可测试性

代码生成时应主动考虑测试。

更容易测试的代码通常具备：

- 输入输出明确。
- 业务逻辑不直接依赖当前时间、随机数、网络、文件系统或全局状态。
- 外部副作用集中在边界层。
- 核心逻辑可以通过普通函数调用验证。
- 错误路径和边界条件可以被单独覆盖。

不推荐在核心逻辑中直接调用外部系统：

```python
def calculate_discount(user_id: str) -> Decimal:
    user = database.get_user(user_id)
    ...
```

更推荐把外部数据作为参数传入：

```python
def calculate_discount(user: User) -> Decimal:
    ...
```

## 状态和副作用

副作用应显式、集中、可替换。

常见副作用包括：

- 写文件
- 写数据库
- 发送 HTTP 请求
- 发布消息
- 打印输出
- 修改全局变量
- 读取当前时间

要求：

- 不要在导入模块时执行重要副作用。
- 不要在全局作用域发起网络请求、读取大型文件或启动后台任务。
- 副作用函数的名称应能体现行为，例如 `save_user`、`send_email`、`write_report`。
- 核心计算函数应尽量保持纯函数。

## 配置处理

配置应集中读取和校验，不要散落在业务逻辑中。

推荐：

- 在启动阶段读取环境变量。
- 使用配置对象传递配置。
- 给配置项提供明确名称和类型。
- 对缺失或非法配置尽早报错。

不推荐：

- 在多个函数中重复调用 `os.getenv()`。
- 在业务逻辑深处读取环境变量。
- 使用魔法字符串和魔法数字。
- 将密钥、token 或密码写入代码。

## 错误处理逻辑

错误处理应服务于调用方理解和恢复。

- 对可预期的业务错误使用明确异常或返回结果类型。
- 对不可恢复错误允许向上抛出，但应保留上下文。
- 不要把所有异常都转换成 `Exception`。
- 不要吞掉异常后返回 `None`，除非调用方明确需要这种语义。
- 对外部系统错误，应区分超时、认证失败、限流、数据不存在和响应格式错误。

## 扩展方式

扩展代码时优先选择局部扩展，而不是全局重构。

好的扩展方式：

- 在现有接口下增加一个小函数。
- 在现有模型中添加明确字段。
- 在现有分支中增加一个清晰 case。
- 抽出重复且稳定的逻辑。
- 为新行为补充测试。

需要谨慎的扩展方式：

- 改变公共函数签名。
- 改变数据模型含义。
- 引入全局注册表。
- 引入继承层级。
- 重排目录结构。
- 替换已有框架或工具链。

## 抽象原则

只有当抽象能降低真实复杂度时才引入抽象。

适合抽象的信号：

- 同一逻辑已经在多个地方稳定重复。
- 调用方确实需要统一接口。
- 抽象后能减少分支和重复状态。
- 抽象边界能用清晰名称描述。

不适合抽象的信号：

- 只是为了预测未来需求。
- 只是为了让代码看起来更高级。
- 抽象名称含糊，例如 `Manager`、`Handler`、`Processor`，但职责不清。
- 抽象后调用链更长，理解成本更高。

## 并发和异步

只有在明确需要并发 I/O、后台任务或吞吐能力时才使用异步或并发。

- 不要把同步代码无意义地改成 `async`。
- 使用 `async` 时，调用链应保持一致，避免同步和异步混杂。
- 不要在 async 函数中直接执行阻塞 I/O。
- 并发任务应有超时、取消和错误处理。
- 共享状态需要明确同步策略。

## 性能优化

性能优化应基于事实，而不是猜测。

优先顺序：

1. 写出清晰正确的代码。
2. 用测试保证行为。
3. 用日志、指标或 profiling 找到瓶颈。
4. 只优化瓶颈路径。
5. 用注释说明不直观的优化原因。

不要为了微小性能收益引入复杂缓存、全局状态或难以维护的数据结构。

## 生成代码时的检查清单

完成代码后，应检查：

- 是否使用了项目已有风格和工具链。
- 是否能用更简单的函数表达。
- 是否存在不必要的类、继承或抽象。
- 是否存在深层嵌套和过长函数。
- 数据结构是否清晰且类型明确。
- 外部输入是否在边界处校验。
- 副作用是否集中且可测试。
- 错误处理是否保留上下文。
- 新增依赖是否必要，并通过 `uv` 管理。
- 是否有对应的测试或可验证命令。
